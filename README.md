# G's課題 提出README （従業員登録CRUD）


## ① 課題名 ／ 主要ファイル

**従業員登録のCRUD ― 「フォーム → 検査 → INSERT → SELECT → 一覧表示」を、本番サービスの職員マスタで実地にやった**

主要ファイル（ChildcareOS / `feature/home-kpi` ブランチ）:


| `app/Models/Emploee.php` | テーブルとPHPを繋ぐモデル（保存を許可する列 `$fillable`） | テーブルの列定義の対応物 |
| `app/Http/Requests/Setting/StoreEmploeeRequest.php` | **入力検査（バリデーション）** | ★授業に無い。実サービスの肝 |
| `app/Http/Controllers/Setting/EmploeeController.php`（`store`） | フォーム入力→DB保存（INSERT） | insert.php |
| `app/Http/Controllers/Setting/EmploeeController.php`（`index`） | 条件付きSELECT＋表示用に整形 | select.php |


## ② 課題内容（どんな作品か）
職員データで、それを画面から登録するのがこの「従業員登録」機能。

**① 画面のフォームに職員情報を入力**（名前・資格・在籍開始日・配属施設 など）
→ **② 「追加」ボタンで Laravel がPOSTを受け取る**
→ **③ まずバリデーションが走る**（必須か・型は正しいか・資格は決められた一覧のものか）
→ **④ 検査を通った値だけで INSERT**（職員を1行登録）
→ **⑤ 一覧画面を開くと SELECT で全件取得**
→ **⑥ `@foreach` で1人ずつ表の行にして表示**
---

## ⑤ 工夫した点・こだわった点
授業の `insert.php` は、フォームから来た値をそのままINSERTします。だけどAIと壁打ちしているとバリデーションの観点が出てきた。なので本物のサービスは知らない人が大勢触るので、空欄も、変な文字も、想定していない項目も普通に飛んできます。それをDBに入れる前に止める必要があった。
しかも Laravel は、コントローラ本体（`store()`）が動く前にこの検査を自動で走らせます。検査に通らなければ `store()` 自体が実行されず、エラーをつけてフォームに戻る。この構造が良くできてるなと思いました。

## ⑥ 難しかった点・次回トライしたいこと

難しかったのは、**「rules() の検査」と「validated() の選別」を、バリデーションでひとくくりにしていた。実際は「変な値を弾く」のと「余計な列を混ぜない」のは別の仕事で、ここを分けて理解できた。
あとはDB登録するときにすべて成功しないと処理を巻き戻す処理。

## ⑦ フリー項目（感想）

一番の気づきは、「バリデーション」が、実サービスでは一番大事だということ。動かすだけなら検査は無くても動きます。でも知らない人が触る本物のサービスでは、空欄も想定外の値も余計な項目も必ず来るので、DBに入れる前に止める仕組みが無いとデータがすぐ壊れる。
